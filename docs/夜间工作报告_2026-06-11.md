## 二、本轮新发现并修复的 bug

### 1.「治疗意见及措施」字段映射指向不存在的章节（中危）
- 现象：LLM 质量建议给出的"治疗意见及措施"父段修复文本被静默丢弃（上一轮 E2E 里 chip
  不出现的根因）。
- 根因：映射表里它指向【处理意见】——渲染器从不输出这个独立章节（门诊模板只有
  【治疗意见及措施】+ 四个子行）。同时 treatment_method/treatment_plan 等英文键也指向
  不存在的独立章节。
- 修复：映射改指真实章节（复合段合并保护下安全写入）；英文键补齐行级映射。

### 2. 行级写入"章节缺失即静默放弃"的设计缺陷（中危，治本）
- 同一字段在不同病历类型里粒度不同：门诊"注意事项"是【治疗意见及措施】的子行，
  住院"日常病程记录"里它是独立章节。原行级写入找不到目标章节直接放弃。
- 修复：行级写入无效时回退到章节级定位，两种模板都能正确写入。+3 个回归测试。

### 3. PACS 旧版数据点击无反馈（低危 UX）
- R1 迁移前的旧检查（无 study_instance_uid）后端按设计返回 410，但前端只弹泛化的
  "加载失败"。改为透传后端原因："该检查为旧版本数据，已不支持查看"。浏览器实测通过。

### 4. antd Spin 用法警告 ×3
- lazyFallback / 应用启动 / 嵌入加载页的 Spin tip 不在嵌套模式下不生效且报警告，已修。

## 三、主动优化（参考业界成熟方案）

### 1. LLM 输出数值真实性守卫（医疗安全，最重要的一项）
- 背景：prompt 红线是软约束，昨晚实测 LLM 补全仍会编造"默认正常"生命体征。
- 方案：业界标准的 **output guardrail（输出侧确定性后校验）**——新增
  backend/app/services/ai/output_guards.py：生命体征 token（T/P/R/BP/SpO2）中的数值
  必须能在医生录入的原始数据里找到字面出处，查无出处的整个 token 剔除、只留描述文字。
  宁可误删（医生可手填）不可漏放（编造数值可能被直接签发）。
- 接入：批量补全链路；+8 个单测覆盖编造/有出处/混合/纯描述/非体征数字等边界。

### 2. AI 建议接口降级标记
- 原先追问/检查/诊断建议接口在 AI 故障时静默返回空列表，医生分不清"真没建议"和
  "服务挂了"。后端 3 处异常兜底加 degraded=true，前端 3 处消费点改为明确报
  "AI 服务暂时不可用"。

### 3. qcFieldMaps.ts 858 行超标文件拆分（规范欠债）
- 拆为 4 个职责单一的模块 + 桶文件 re-export，**全项目 import 路径零改动**：
  qcFieldConstants(209行 映射表) / qcFieldMeta(143行 元数据) /
  recordSectionWriter(302行 写入核心) / recordFieldState(239行 快照撤销定位)。
- 83 个相关测试全过，零回归。

### 4. pacs.py 日志规范统一
- 11 处不合规日志改为 "pacs.action: key=value" 模板（rid 由 RequestIDFilter 自动注入），
  线上排查可按模块+动作聚合。
