## 二、P0 —— 必须立即修（上线阻塞）

### P0-1 桌面 Agent /fill 接口完全没有鉴权 【安全】
- 位置：tools/desktop_agent/http_server.py:119（token 校验还是 TODO，已亲自验证）
- 问题：前端虽然带了 Bearer token，但 Agent 端根本不验。医生电脑上**任何进程**
  （恶意软件、其他网页诱导的本地请求）都能直接 POST http://127.0.0.1:7788/fill
  往 HIS 里写任意内容（改诊断、改过敏史）。CORS 只能挡浏览器，挡不住本地程序。
- /progress WebSocket 的 ?token= 同样没验证。
- 修法：Agent 端解析 Authorization header，验 JWT 签名 + embed claim + 过期时间；
  WS 握手时同样校验。密钥通过 Agent 配置下发或改用后端代验。

### P0-2 embed_token 4 小时过期后前端无任何处理 【嵌入模式】
- 位置：frontend/src/pages/EmbedWorkbenchPage.tsx:43-44 + store/embedStore.ts
- 问题：embedStore 用 sessionStorage 持久化但没存过期时间。token 过期后页面刷新
  仍然"看起来正常"，随后所有 API 全部 401，医生只会看到莫名其妙的报错。
- 修法：embedStore 存 expires_at，挂载时检查；过期则提示"嵌入会话已过期，
  请重新从 HIS 触发"。

### P0-3 全局异常响应泄露内部异常类型 【安全】
- 位置：backend/app/main.py:90
- 问题：500 响应里带 "error_type": type(exc).__name__，向攻击者暴露内部实现，
  不符合等保 2.0 三级对错误信息的要求。
- 修法：删掉 error_type 字段，细节只进服务端日志/Sentry。

### P0-4 /auth/check-username 无限速可枚举用户名 【安全】
- 位置：backend/app/api/v1/auth.py:140-152
- 问题：公开端点 + 无速率限制 + 明确返回存在与否 → 可批量枚举全院医生账号，
  再配合弱密码爆破。
- 修法：加 IP 限速（复用 login_limiter）；考虑响应不区分存在性。

### P0-5 PACS 报告发布/删除多步写无事务保护 【数据一致性】
- 位置：backend/app/api/v1/pacs.py:1143-1185（save_report+publish）、:1345 附近（delete）
- 问题：report 与 study 两个对象的状态修改不在同一事务里；删除时先删 DB 再调
  Orthanc，顺序与失败补偿都没设计 → 中途失败会留下"报告已发布但影像状态没动"
  之类的不一致医疗数据。
- 修法：用 async with db.begin() 包裹；删除改为 DB 成功后再删 Orthanc，
  Orthanc 失败记 error 日志留人工清理。

## 三、P1 —— 一周内修（高优先级）

### P1-1 依赖已知漏洞（前后端都有）
- 后端 python-jose==3.3.0：JWT 签名库，有 CVE-2024-33663（算法混淆）和
  CVE-2024-33664（解码炸弹 DoS）两个已知漏洞，升级到 ≥3.4（建议 3.5）。
  已确认 app/core/security.py:25 在用它签发/校验 token。
- 前端 axios 1.7.7：npm audit 报 high，一串原型污染/代理绕过/ReDoS CVE，
  升级到最新 1.x。
- npm audit 共 15 个漏洞（3 high / 12 moderate），babel 插件与 fast-uri 的
  high 直接 `npm audit fix` 可修。
- requirements.txt 里 pymupdf / numpy / Pillow / alibabacloud-bssopenapi 没钉版本，
  构建不可复现且有供应链风险，建议钉死。

### P1-2 CORS 过宽 + 缺安全响应头 【安全】
- 位置：backend/app/main.py:61-67
- allow_methods=["*"]、allow_headers=["*"] 收窄到实际用的；
  补 X-Content-Type-Options / X-Frame-Options / HSTS / CSP 中间件。

### P1-3 嵌入会话创建"患者+接诊"两步写无事务 【嵌入模式】
- 位置：backend/app/api/v1/embed.py:89-102
- 患者建好后接诊失败 → 孤儿患者记录。包进一个事务。

### P1-4 复合段拆分正则边界没有测试覆盖 【嵌入模式】
- 位置：frontend/src/components/workbench/recordEditor/RecordEditorToolbar.tsx:192-221
- AI 输出英文冒号 / 子项间空行 / 子项缺失时，正则可能整段匹配失败，兜底把
  整个复合段当一个字段推给 Agent，而 Agent 的字段映射表里没有父字段 → 填入失败。
- 修法：补 4 类边界单测（英文冒号、空行分隔、子项缺失、单子项）。

### P1-5 desktopAgent WebSocket 无错误处理 + AutoFillButton 连点竞态 【嵌入模式】
- frontend/src/services/desktopAgent.ts:134-151：没有 onerror/onclose，Agent 宕机时
  医生卡在"准备开始…"直到 60s 超时；WS 也没显式 close。
- frontend/src/components/embed/AutoFillButton.tsx：handleAutoFill 开头没有
  `if (filling) return` 互斥（已亲自验证），快速连点/程序触发可并发多个 fill。

### P1-6 AI 调用链健壮性
- record_gen_v2_service.py:37-67：JSON 重试 3 次无指数退避，3 次全失败直接抛给前端。
- ai_generation.py:86-124：流式生成锁 TTL 120s，前端断网后医生要干等 2 分钟才能重新生成。
- record_prompts.py / ai_voice.py:371：患者姓名等字段直接 format 进 prompt，
  没清洗换行/控制字符，存在 prompt 注入面。coalesce_field 里加
  `replace('\n',' ').replace('\r',' ')` 级别的清洗。

### P1-7 文件行数超标 12 个（违反 ≤300/400/250 规范）
